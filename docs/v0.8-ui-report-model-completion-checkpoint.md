# v0.8 UI Report Model Completion Checkpoint

## Purpose

This checkpoint records the completed v0.8 UI-consumable report model boundary after PR #84, PR #85, and PR #86.

v0.8 moves the reporting / diagnostic UX path from formatter-only output toward a stable data model that downstream UI layers can consume without reparsing formatted strings.

## Completed scope

The v0.8 UI report model path now includes:

- the v0.8 UI-consumable report model entry boundary,
- a minimal `ReportModel` type,
- a minimal `ReportSection` type,
- a minimal `ReportRow` type,
- a generated-target comparison report adapter,
- a plain-text fallback formatter for report models,
- tests for section ordering,
- tests for row metadata preservation,
- tests for missing generated-target status handling,
- tests for unchanged expected reward baseline.

## Source API

The UI-consumable report model is represented by:

```ts
export type ReportModel = {
  kind: string;
  title: string;
  sections: ReportSection[];
};

export type ReportSection = {
  id: string;
  title: string;
  rows: ReportRow[];
};

export type ReportRow = {
  id: string;
  label: string;
  plainText: string;
  tex?: string;
  className?: string;
  status?: ReportRowStatus;
  metadata?: Record<string, unknown>;
};
```

The initial adapter is:

```ts
generatedTargetComparisonReportToReportModel(report)
```

The plain-text fallback formatter is:

```ts
formatReportModelPlainText(report)
```

## Generated-target comparison model

The generated-target comparison report can now be represented as structured UI data.

The model preserves:

- report kind,
- report title,
- summary section ordering,
- row section ordering,
- row labels,
- row plain text,
- row status,
- class metadata,
- machine-readable metadata for `from`, `explicitTo`, `generatedTo`, missing status, and comparison status.

This lets UI consumers render summary / comparison views without parsing Markdown or plain text.

## Plain-text fallback

`formatReportModelPlainText()` emits fallback text from the report model's existing `plainText` rows.

The fallback formatter does not replace the structured model. It provides a simple output path for environments that need text-only display, logging, snapshot-style comparison, or copy-friendly output.

## Preserved runtime boundaries

This checkpoint does not change runtime semantics.

The following remain fixed:

- solver target is `transition.to`,
- contribution target is `transition.to`,
- `generatedTo` is validation / diagnostics / summary / comparison report data only,
- `generatedTo` is not used as solver target,
- `generatedTo` is not used as contribution target,
- runtime target substitution remains unimplemented and out of scope,
- `generated_candidate_as_solver_target` remains rejected as runtime policy,
- expected reward baseline remains unchanged.

## Expected reward baseline

The representative expected reward baseline remains:

```text
V(position=3) = 0
V(position=2) = 1
V(position=1) = 1.5
V(position=0) = 2.25
```

The UI report model work does not alter solver output or reward calculation.

## Completion boundary

The v0.8 UI report model work is complete when:

- a minimal UI-consumable report model exists,
- generated-target comparison report data can be converted into that model,
- generated-target comparison metadata remains machine-readable,
- plain-text fallback can be generated from the model,
- existing formatter behavior is not removed,
- runtime target substitution remains out of scope,
- expected reward baseline remains unchanged.

This checkpoint satisfies that boundary.

## Next boundary

The next safe step is a v0.9 reporting integration boundary.

Potential v0.9 work may connect additional existing diagnostic / summary report paths to `ReportModel`, but it must not connect `generatedTo` to solver or contribution execution paths.
