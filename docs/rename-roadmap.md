# Rename roadmap

Rename is required for an international-facing project, but code-level rename remains blocked until the public output boundary and solver target policy are stable.

## Phases

1. Naming design: define international technical names, Japanese names, and codenames.
2. Public wording alignment: use Outcome Return Function, Forward Outcome Engine, and Inverse Outcome Estimator in public docs.
3. Output boundary stabilization: stabilize expected reward, contribution, graph diagnostics, and JSON output shapes.
4. Code-level rename: rename helpers, tests, exports, APIs, and file paths in controlled batches.

## Current boundary

Current work may update docs wording and glossary files.

Current work must not rename existing exports, public APIs, package identity, or file paths.

Kiyotan and Seikatan remain codenames. They should be kept as aliases while international technical names are introduced.
