## Generic PoC boundary to preserve

- A PoC describes a state-transition model before it describes a machine-specific feature.
- State movement remains represented through transitions and explicit target semantics.
- Solver target remains `transition.to`.
- Contribution target remains `transition.to`.
- Generated target information remains diagnostic/reporting data.
- Reporting adapters may summarize generated target validation results.
- Runtime target substitution remains outside the current boundary.

## Inputs to separate before Beast King PoC

Beast King PoC seed work must separate these categories before fixture expectations are locked:

1. Confirmed inputs
   - Values copied from a confirmed source or explicit project decision.
   - These may be used directly in seed documentation or fixture stubs.

2. Assumptions
   - Values introduced only to make a minimum PoC executable.
   - These must be named as assumptions and kept replaceable.

3. Unknowns
   - Values not yet available.
   - These must not be silently filled into production behavior.

4. Out-of-scope items
   - UI framework behavior.
   - CSS behavior.
   - Android native behavior.
   - TeX rendering changes.
   - Runtime generated-target substitution.
