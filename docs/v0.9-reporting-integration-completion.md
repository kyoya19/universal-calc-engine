# v0.9 Reporting Integration Completion Checkpoint

## Purpose

This document records the completion checkpoint for the first v0.9 reporting integration step.

v0.9 started from the v0.8 `ReportModel` foundation and added one additional existing reporting path to the structured report model layer without changing solver or contribution runtime semantics.

## Completed implementation

The first v0.9 implementation added a `ReportModel` adapter for generated-target solver gate summaries.

The added adapter is:

- `generatedTargetSolverGateResultSummaryToReportModel()`

It converts `GeneratedTargetSolverGateResultSummary` into a UI-consumable `ReportModel`.

## Covered source path

The covered existing reporting path is:

- `summarizeGeneratedTargetSolverGateResult()`
- `formatGeneratedTargetSolverGateResultSummary()`

The new adapter uses the stable summary source data directly. UI consumers do not need to reparse the existing plain text formatter output to obtain row identity, status, or metadata.

## Report model structure

The generated-target solver gate summary report model uses:

- `kind: generated_target_solver_gate_summary`
- `title: Generated Target Solver Gate Summary`
- one `summary` section

For accepted summaries, the stable rows are:

- `accepted`
- `edgeCount`
- `generatedTargetReadyEdgeCount`

For rejected summaries, the stable rows are:

- `accepted`
- `edgeCount`
- `generatedTargetReadyEdgeCount`
- `rejectionCode`
- `rejectionType`
- `rejectionMessage`

## Plain text fallback

`formatReportModelPlainText()` remains the generic fallback formatter.

For the generated-target solver gate summary adapter, the fallback body remains aligned with the existing `formatGeneratedTargetSolverGateResultSummary()` output under the report title and summary section heading.

## Tests added

The v0.9 implementation added tests for:

- accepted gate summary section ordering,
- accepted gate summary row ordering,
- accepted gate summary row status and metadata,
- rejected gate summary row ordering,
- rejected gate summary rejection metadata,
- fallback plain text alignment with the existing formatter body,
- unchanged generated-target gate behavior,
- unchanged expected reward baseline.

## Runtime boundary confirmed

This checkpoint keeps the following boundaries unchanged:

- solver target remains `transition.to`,
- contribution target remains `transition.to`,
- `generatedTo` remains validation / diagnostics / summary / comparison report data only,
- `generatedTo` is not used as a solver target,
- `generatedTo` is not used as a contribution target,
- runtime target substitution remains unimplemented,
- `generated_candidate_as_solver_target` remains rejected as runtime policy,
- existing generated-target comparison report model remains compatible,
- existing formatter output remains available.

## Expected reward baseline confirmed

The representative expected reward baseline remains:

```text
V(position=3) = 0
V(position=2) = 1
V(position=1) = 1.5
V(position=0) = 2.25
```

The v0.9 reporting integration step does not alter solver output or reward calculation.

## Completion status

v0.9 reporting integration entry required at least one additional existing reporting path beyond the initial generated-target comparison adapter to be represented as a structured `ReportModel` with tests for ordering, metadata, fallback text, and unchanged runtime behavior.

That requirement is now satisfied by the generated-target solver gate summary adapter.

## Recommended next entry

The next safe entry is v1.0 reporting surface consolidation.

Recommended scope:

- keep all v0.9 runtime boundaries unchanged,
- document a small public reporting surface for current adapters,
- avoid UI framework work,
- avoid CSS or Android native implementation,
- avoid runtime target substitution,
- avoid connecting `generatedTo` to solver or contribution targets,
- add only documentation or thin barrel-export tests unless a second structured adapter is selected explicitly.
