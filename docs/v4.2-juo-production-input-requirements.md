# v4.2 Juo production input requirements

## Status

Requirements checkpoint for converting the v4.1 production-input planning entry into an implementation-ready checklist.

Current main baseline:

- #142 Document v4.1 Juo production input planning entry
  - merge commit: `3e56ae63566121ce5f911303348809438fbbb3ef`

## Purpose

This document records the concrete requirements that must be satisfied before Beast King / Juo production data is inserted into fixtures or production source.

The checkpoint remains documentation-only. It does not introduce real probabilities, real rewards, runtime target substitution, machine-specific reporting, formatter changes, contribution changes, TeX changes, CSS changes, Android native changes, or UI changes.

## Production-input gate

Juo must remain placeholder-only until all required fields below are either completed or explicitly marked as deferred.

Allowed statuses:

- `required`: must be supplied before implementation.
- `stub`: may be represented by an explicit placeholder in a fixture or document.
- `deferred`: intentionally outside the first production-input pass.
- `excluded`: explicitly not modeled in the first production-input pass.

Unmarked omissions are not allowed.

## Validation requirements

Validation must be split into generic and Juo-specific layers.

Generic validation requirements:

- State identifiers are unique.
- Start state exists.
- Terminal states are explicit.
- Transition source states exist.
- Transition target states exist.
- Transition probability references exist.
- Transition reward references exist.
- Placeholder values are marked.

Juo-specific validation requirements:

- Machine scope is present.
- Included and excluded routes are documented.
- Ambiguous states are marked.
- Probability source notes are present.
- Reward source notes are present.
- Accounting exclusions are present.

Deferred validation requirements:

- Real probability normalization enforcement.
- Real reward unit arithmetic enforcement.
- Machine-specific report row validation.
- Runtime target substitution validation.

## Regression baseline requirements

The first production-input implementation should use the following regression order:

1. Requirements document exists.
2. Fixture-only state table stub exists.
3. State identity validation passes.
4. Terminal-state validation passes.
5. Transition reference validation passes.
6. Probability placeholder validation passes.
7. Reward placeholder validation passes.
8. Generic output shape remains unchanged.
9. Sugoroku expected reward baseline remains unchanged.
10. Juo expected reward remains unresolved until real probabilities and rewards are approved.

## First implementation constraints

The first implementation after this requirements checkpoint should be fixture-only unless a separate implementation checkpoint approves production source work.

Allowed first implementation changes:

- Add a Juo production-input fixture stub.
- Add tests that validate shape and unresolved placeholders.
- Add documentation connecting the fixture stub to this requirements document.

Disallowed first implementation changes:

- Real Beast King / Juo probability values.
- Real Beast King / Juo reward values.
- Expected value assertion for Juo.
- Report adapter changes.
- Formatter changes.
- Runtime target substitution.
- TeX / CSS / Android native / UI rendering changes.
