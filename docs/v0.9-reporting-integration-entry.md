# v0.9 Reporting Integration Entry

## Purpose

This document defines the entry boundary for v0.9 reporting integration work.

v0.8 completed the minimum UI-consumable report model path. v0.9 should now integrate additional existing reporting / diagnostic outputs into `ReportModel` without changing solver or contribution runtime semantics.

## Goal

The v0.9 goal is to expand report-model coverage across existing reporting paths.

The work should let downstream UI consumers render more diagnostic and summary output from stable structured data instead of reparsing formatted Markdown or plain text.

## Starting point

The current v0.8 foundation includes:

- `ReportModel`,
- `ReportSection`,
- `ReportRow`,
- `ReportRowStatus`,
- `generatedTargetComparisonReportToReportModel()`,
- `formatReportModelPlainText()`,
- tests for section ordering,
- tests for metadata preservation,
- tests for missing generated-target handling,
- tests for unchanged expected reward baseline.

## In scope

v0.9 may add report-model adapters for existing outputs such as:

- generated-target summaries,
- generated-target validation diagnostics,
- generated-target comparison details beyond the initial adapter,
- existing reporting / diagnostic formatter outputs that already have stable source data,
- Android-friendly report sections that can reuse existing row-level display metadata.

The first v0.9 implementation PR should select one existing path only.

## Out of scope

v0.9 entry work does not include:

- runtime target substitution,
- generated-target execution,
- replacing solver targets with generated targets,
- replacing contribution targets with generated targets,
- changing expected reward calculations,
- changing the solver baseline,
- UI framework components,
- CSS files inside `packages/core`,
- Android-specific native code.

## Required runtime boundaries

The following remain fixed:

- solver target is `transition.to`,
- contribution target is `transition.to`,
- `generatedTo` is validation / diagnostics / summary / comparison report data only,
- `generatedTo` is not a solver target,
- `generatedTo` is not a contribution target,
- runtime target substitution is not implemented,
- `generated_candidate_as_solver_target` remains rejected as runtime policy,
- expected reward baseline remains unchanged.

## Expected reward baseline

The representative expected reward baseline remains:

```text
V(position=3) = 0
V(position=2) = 1
V(position=1) = 1.5
V(position=0) = 2.25
```

v0.9 reporting integration must not alter solver output or reward calculation.

## Recommended first implementation step

The safest first implementation step is to add a `ReportModel` adapter for one existing diagnostics or summary path that already has stable source data.

The first implementation PR should:

- add one adapter only,
- add tests for section ordering,
- add tests for row metadata preservation,
- add tests for fallback plain text output when applicable,
- keep existing formatter output compatible,
- avoid runtime solver or contribution target changes,
- keep the expected reward baseline unchanged.

## Completion target

v0.9 can be considered complete when at least one additional existing reporting path beyond the initial generated-target comparison adapter is represented as a structured `ReportModel`, with tests covering ordering, metadata, fallback text, and unchanged runtime behavior.
