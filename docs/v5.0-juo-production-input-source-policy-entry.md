## Purpose

This phase defines the policy for accepting real Beast King / Juo production input sources before any real probability or reward values are inserted.

The phase must decide how source, citation, unit, inference, and formula metadata are represented so that later production-input rows can distinguish confirmed facts from derived or unresolved values.

## Source policy areas to define

The later source policy must cover at least the following areas before any real value insertion:

- Machine scope.
- Source type classification.
- Citation requirements.
- Source priority and conflict handling.
- Probability unit representation.
- Reward unit representation.
- Route inclusion and exclusion.
- Inferred value representation.
- Formula value representation.
- Unknown or unavailable value representation.
- Human-entered transcription handling.
- Normalization and arithmetic traceability.
- Change-review requirements for replacing placeholder/stub rows with source-backed rows.

## Source type classification

The source policy should distinguish source categories before accepting a row as production input.

Minimum categories:

- `primary`: official manual, official specification, manufacturer-published material, or other direct official source.
- `secondary`: reputable compiled source that cites or reproduces machine values.
- `observed`: value derived from measured play data, screenshots, counters, or other observation.
- `inferred`: value derived from calculation, reverse solving, complement arithmetic, or consistency constraints.
- `assumption`: explicitly assumed value used only for modelling or sensitivity analysis.
- `unknown`: value not yet known or not yet accepted.
- `excluded`: value intentionally out of model scope.

## Citation requirements

Every later real production-input value must carry citation metadata or an explicit non-citation status.

Minimum citation metadata:

- Source type.
- Source title or identifier.
- Location inside the source where practical.
- Retrieval or observation date when the source can change.
- Human transcription status when manually copied.
- Confidence status.
- Notes for conflicts, exclusions, or inferred values.

A row must not silently move from `stub` to executable value without source metadata.

## Unit policy requirements

Probability and reward values must not be accepted unless their unit meaning is explicit.

Probability rows must specify whether the value is represented as one of the following:

- Ratio.
- Percent.
- Fraction.
- Count over denominator.
- Conditional probability.
- Complement-derived probability.
- Formula-derived probability.

Reward rows must specify whether the value is represented as one of the following:

- Payout balls.
- Net balls.
- Medals.
- Currency.
- Points.
- Time.
- Composite expected reward.
- Formula-derived reward.

## Inference and formula policy requirements

The later policy must preserve a strict distinction between direct values, inferred values, and formula values.

Required distinctions:

- A direct value is copied from an accepted source.
- An inferred value is derived from one or more accepted or provisional inputs.
- A formula value stores the expression and its dependencies, not only the numeric result.
- A provisional value must not be treated as a confirmed production value.
- A sensitivity-analysis assumption must not become a production value without an explicit promotion step.

## Conflict handling requirements

When multiple sources disagree, later production-input rows must not choose a value silently.

At minimum, the policy must require:

- Recording the conflicting values.
- Recording each source category.
- Recording the selected value and selection reason.
- Preserving rejected values as notes or separate non-executable candidates where practical.
- Marking unresolved conflicts as non-executable.

## Acceptance gates for later real value insertion

A later PR that inserts real Beast King / Juo production values must satisfy all of the following gates or explicitly defer the row:

- The machine scope is fixed.
- The route inclusion and exclusion list is fixed.
- Each executable probability row has source and unit metadata.
- Each executable reward row has source and unit metadata.
- Inferred values identify their dependencies.
- Formula values preserve their expression.
- Unknown values remain non-executable.
- Conflicts are recorded before selection.
- The review boundary states whether the change is still source-policy-only or value-inserting.
