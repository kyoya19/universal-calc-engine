# v4.1 Juo production input planning entry

## Status

Entry checkpoint for planning the Beast King / Juo production-input phase after the placeholder-boundary phase was closed.

Current main baseline:

- #141 Document v4.0 Juo placeholder-boundary phase completion
  - merge commit: `b302ea68cae0e5174939baf7e543eb334cd5b6ac`

## Purpose

This checkpoint defines what must be specified before Juo leaves placeholder-only status.

The goal is not to introduce production behavior yet. The goal is to make the production-input boundary explicit before any production source, fixture expansion, report adapter, formatter, contribution output, TeX, CSS, Android native, or UI rendering change is introduced.

## Planning-only boundary

The v4.1 entry is documentation-only and preserves the v4.0 boundary:

- No production source change.
- No fixture data expansion.
- No Juo production state graph.
- No Juo production probability insertion.
- No Juo production reward insertion.
- No Juo-specific report adapter.
- No Juo-specific report kind.
- No Juo-specific report title.
- No formatter change.
- No contribution output change.
- No runtime target substitution.
- No TeX / CSS / Android native / UI rendering change.
- Existing Sugoroku expected reward baseline remains unchanged.

## Required production-input groups

Before implementation begins, the following input groups must be defined or explicitly stubbed.

### 1. Machine scope

Required decisions:

- Target machine name policy.
- Whether the phase models Beast King / Juo as a whole machine or a narrowed sub-problem.
- Whether the first production graph models only one bonus / mode / route class.
- Whether incomplete routes are allowed as explicit stubs.

Output expected before implementation:

- A short machine-scope statement.
- A list of included routes.
- A list of excluded routes.
- A list of known stubs.

### 2. State model

Required decisions:

- State identity format.
- State display label format.
- Terminal-state definition.
- Start-state definition.
- Whether transient helper states are allowed.
- Whether named states may map to multiple observed machine situations.

Output expected before implementation:

- State table.
- Start-state row.
- Terminal-state rows.
- Any helper-state rows.
- Any ambiguous-state notes.

### 3. Transition model

Required decisions:

- Transition identity format.
- Transition source and target state policy.
- Whether transition labels are machine-facing or model-facing.
- Whether transition grouping is allowed.
- Whether generated targets remain diagnostic-only.

Output expected before implementation:

- Transition table.
- Source state.
- Target state.
- Probability reference.
- Reward reference.
- Diagnostic target metadata policy.

### 4. Probability model

Required decisions:

- Probability source policy.
- Whether exact fractions are required.
- Whether decimal values are allowed.
- Whether inferred probabilities are allowed.
- Whether unresolved probabilities can be explicit placeholders.
- Whether probability normalization is enforced at fixture level or validation level.

Output expected before implementation:

- Probability table.
- Source / citation note.
- Unit and denominator note.
- Placeholder marker for unresolved values.
- Normalization rule.

### 5. Reward model

Required decisions:

- Reward unit policy.
- Whether reward means payout, net payout, expected coins, expected value, or another quantity.
- Whether time, exchange rate, and cost are excluded from the first production graph.
- Whether zero reward on terminal states is fixed.
- Whether rewards may be formulas or only literals.

Output expected before implementation:

- Reward table.
- Reward unit definition.
- Formula policy.
- Excluded accounting dimensions.
- Placeholder marker for unresolved values.

### 6. Unit model

Required decisions:

- Probability unit.
- Reward unit.
- Count unit.
- Time unit.
- Money / exchange unit, if any.
- Whether units are human-readable metadata only or validation inputs.

Output expected before implementation:

- Unit table.
- Unit-to-field mapping.
- Explicit exclusions.

### 7. Target semantics

Required decisions:

- Solver target remains `transition.to` unless changed by a separate production-facing decision.
- Contribution target remains `transition.to` unless changed by a separate production-facing decision.
- `generatedTo` remains diagnostic-only unless a new role is explicitly introduced.
- Runtime target substitution remains out of scope unless separately designed.

Output expected before implementation:

- Target semantics statement.
- Any proposed change and its migration impact.
- Regression tests required for any change.

### 8. Report identity

Required decisions:

- Whether Juo stays on generic `generated_target_comparison` report output.
- Whether Juo gets a dedicated report adapter.
- Whether Juo gets a dedicated report kind.
- Whether Juo gets a dedicated report title.
- Whether machine-specific rows are introduced.

Output expected before implementation:

- Report identity decision.
- Report row schema decision.
- Formatter impact decision.
- Compatibility notes for existing generic output.

### 9. Validation rules

Required decisions:

- Required state fields.
- Required transition fields.
- Probability normalization rule.
- Reward-unit validation rule.
- Unresolved placeholder validation rule.
- Machine-specific rule scope.

Output expected before implementation:

- Validation rule list.
- Which rules are generic.
- Which rules are Juo-specific.
- Which rules are deferred.

### 10. Regression baseline strategy

Required decisions:

- Whether the first implementation adds tests before source changes.
- Whether production input starts as fixture-only.
- Whether output shape tests are added before expected reward tests.
- Whether existing Sugoroku baseline is always rechecked.

Output expected before implementation:

- Test sequence.
- Baseline files.
- Expected affected modules.
- Explicit non-affected modules.

## Recommended implementation order after planning

The production-input phase should proceed only after the above groups are documented.

Recommended order:

1. Add Juo production-input requirements document.
2. Add a fixture-only production state table stub.
3. Add state identity and terminal-state validation tests.
4. Add transition shape validation tests.
5. Add probability placeholder validation tests.
6. Add reward placeholder validation tests.
7. Add generic output-shape regression tests.
8. Only then decide whether production source changes are required.

## Explicit non-goals for v4.1

- No implementation of Beast King / Juo expected value.
- No insertion of real machine probabilities.
- No insertion of real machine rewards.
- No machine-specific report adapter.
- No machine-specific formatter branch.
- No runtime target substitution.
- No change to existing generic target semantics.
- No change to existing Sugoroku expected reward baseline.
