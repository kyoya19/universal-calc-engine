# v0.8 UI Consumable Report Model Entry

## Purpose

This document defines the entry boundary for v0.8 UI-consumable report model work.

v0.7 completed the reporting / diagnostic UX checkpoint and the Android-oriented value-function display boundary. v0.8 should now move from formatter-only / display-row outputs toward report-model structures that UI layers can consume directly.

## Goal

The v0.8 goal is to provide stable report data structures for downstream UI consumers.

The model should let a UI render diagnostic and comparison information without reparsing formatted strings.

## In scope

v0.8 may introduce report-model structures for:

- report identity and report kind,
- ordered sections,
- ordered rows within each section,
- machine-readable row metadata,
- human-readable fallback text,
- optional TeX display fragments,
- severity or status labels for diagnostics,
- generated-target summary and comparison views,
- Android-friendly display rows.

The model may be used by formatters, but it should not be limited to formatted Markdown / text output.

## Initial candidate shape

A UI-consumable report model may use a shape similar to:

```ts
export type ReportModel = {
  kind: string;
  title: string;
  sections: ReportSection[];
};

export type ReportSection = {
  id: string;
  title: string;
  rows: ReportRow[];
};

export type ReportRow = {
  id: string;
  label: string;
  plainText: string;
  tex?: string;
  className?: string;
  status?: 'ok' | 'warning' | 'rejected' | 'info';
  metadata?: Record<string, unknown>;
};
```

This is an entry candidate only. The implementation PR should choose the smallest stable subset that matches existing report output.

## Required boundaries

v0.8 must preserve all runtime target boundaries from v0.7.

The following remain fixed:

- solver target is `transition.to`,
- contribution target is `transition.to`,
- `generatedTo` is validation / diagnostics / summary / comparison report data only,
- `generatedTo` is not a solver target,
- `generatedTo` is not a contribution target,
- runtime target substitution is not implemented,
- `generated_candidate_as_solver_target` remains rejected as runtime policy,
- expected reward baseline remains unchanged.

## Out of scope

v0.8 entry work does not include:

- runtime target substitution,
- generated-target execution,
- replacing contribution targets with generated targets,
- changing expected reward calculations,
- changing the solver baseline,
- UI framework components,
- CSS files inside `packages/core`,
- Android-specific native code.

## Acceptance criteria for the first implementation PR

The first implementation PR should be small and should satisfy these criteria:

- introduce a minimal report-model type or builder,
- include tests for ordering and preserved metadata,
- keep existing text / Markdown formatters compatible,
- avoid any runtime solver or contribution target changes,
- keep the expected reward baseline unchanged.

## Recommended first implementation step

The safest first implementation step is to add a minimal report-model type and one adapter from an existing generated-target summary or comparison report path.

That creates a UI-consumable structure without changing runtime behavior or removing existing formatter output.
